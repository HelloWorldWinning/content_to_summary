### **Python 变量命名指南摘要**

#### **一、 变量命名的两个维度：好坏的评判标准**

1.  **内容 (Content):**
    *   **核心：** 传递充足且有效的业务信息。
    *   **标准：** 一个好的变量名能让读者仅从名字就理解其用途和含义（例如，`UserName` 远胜于 `N`）。
    *   **平衡：** 需在信息量与变量名长度之间取得平衡，避免过长或过短。

2.  **形式 (Form):**
    *   **核心：** 保证可读性，清晰地分隔多个单词。
    *   **常见写法：**
        *   **驼峰命名法 (CamelCase):** 通过首字母大写来区分单词，分为大驼峰 (`UserName`) 和小驼峰 (`userName`)。
        *   **下划线分隔法 (snake_case):** 使用下划线分隔全小写的单词 (`user_name`)。

#### **二、 PEP 8 推荐命名约定：追求一致性**

*   **核心原则：** 一致性 (Consistency)。统一的命名风格可以减少认知负担，让代码的阅读者通过命名形式快速判断出实体类型（如类、常量、函数等），从而提高编程效率。
*   **具体规则：**
    *   **常量 (Constants):** 所有字母大写，并用下划线分隔 (`UPPERCASE_WITH_UNDERSCORE`)。
    *   **类 (Classes):** 使用大驼峰命名法 (`CamelCase`)。
    *   **其他所有：** 变量 (Variables)、函数 (Functions)、方法 (Methods)、模块 (Modules) 等，均使用全小写加下划线分隔 (`lowercase_with_underscore`)。

#### **三、 下划线的特殊语义和功能**

1.  **`_` (单个下划线):** 作为**占位符**，用于接收在语法上必须存在但后续不会使用的值。
2.  **`_variable` (前置单下划线):** **弱私有** (Weak Private) 约定。它是一种善意的提醒，告诉其他开发者这是一个内部变量或方法，不应从外部直接调用。`from module import *` 不会导入此类变量。
3.  **`__variable` (前置双下划线):** **强私有** (Strong Private) 属性。在类定义中，Python 会对其进行**名称改写 (Name Mangling)**（改为 `_ClassName__variable`），以避免在子类中被意外覆盖。这是一种防止误用的机制，而非安全特性。
4.  **`__variable__` (前后双下划线):** **魔术方法** (Magic Methods / Dunder Methods)。这是 Python 内部专用的命名方式，用户在自定义变量或方法时**绝对不要**使用。

#### **四、 关键避坑指南**

*   **避免覆盖内置函数：** 不要将变量命名为 `list`, `dict`, `str` 等内置函数或类型名。
*   **避免文件名冲突：** 自己的文件名不要与 Python 的标准库模块同名（如 `turtle.py`），以免导致导入混乱。

***

### **核心结论 (Core Point)**

遵循统一的命名约定（如 PEP 8）不仅是为了代码美观，更是通过提升一致性来传递名称之外的额外元信息，从而减少认知负担，提高编程效率。

### **内容总体框架 (Overarching Framework)**

本内容采用**从“是什么”到“怎么做”再到“为什么”的递进式框架**进行讲解。它首先定义了“合法的”变量名，然后从“内容”和“形式”两个维度探讨了“好的”变量名标准，最后聚焦于 Python 社区公认的 PEP 8 约定，解释了其背后的“一致性”核心原则，并补充了下划线用法和命名冲突等高级话题。
